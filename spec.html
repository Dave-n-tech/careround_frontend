2. Auth architecture

authStore (Zustand)

{
  token: string | null,
  refreshToken: string | null,
  user: { userId, hospitalId, role, firstName, lastName, exp } | null,
  setAuth({ token, refreshToken }),    // decodes token → sets user
  clearAuth(),
  isAuthenticated() → boolean
}

Tokens persisted to localStorage under keys cr_token and cr_refresh. Acknowledged tradeoff: localStorage is XSS-readable. Mitigated by strict CSP (no unsafe-inline scripts) and short token TTL (15 min access, 12h refresh).

Axios instance (api/client.js)

  • Base URL: import.meta.env.VITE_API_BASE_URL
  • Request interceptor: reads token from authStore; injects Authorization: Bearer <token>.
  • Response interceptor: on 401, attempts POST /auth/refresh with stored refreshToken once. On success, retries the original request. On failure, clears auth and redirects to /login.
  • A single in-flight refresh is queued so concurrent 401s don't trigger N refreshes (use a module-level promise).

Wrappers

  • <PrivateRoute/>: if !isAuthenticated(), redirect to /login preserving ?from=.
  • <RoleRoute roles=[…]/>: if user.role not in allowed list, render <Forbidden/> (403 view), do not redirect.

Login flow

  1. User submits email + password → POST /auth/login.
  2. On 200, response carries { token, refreshToken }. Call authStore.setAuth(...).
  3. Decode JWT (utils/jwt.js, no verification client-side) to populate user.
  4. Redirect to role's default route (see §3).
  5. On 401, show inline form error.

4. State management

Three Zustand stores. Server data is fetched per-component (or via SWR-style hooks); we don't cache it globally.

authStore

{ token, refreshToken, user, setAuth, clearAuth, isAuthenticated }

notificationStore

{
  toasts: [{ id, kind: 'success'|'error'|'warn'|'info', title, body?, duration? }],
  addToast(t),
  removeToast(id)
}

roundConductorStore (ephemeral)

{
  roundId: string | null,
  step: 0..5,
  patientQueue: string[],   // patient ids in priority order
  currentIndex: number,
  reviews: { [patientId]: PatientReview },   // unsaved local drafts
  postRoundTasks: { [patientId]: CareTaskDraft[] },
  setRound(round), setStep(n), setIndex(n),
  saveReviewLocal(pid, data), saveTaskLocal(pid, task),
  reset()
}

Persisted to sessionStorage so refresh during a round resumes at the same patient. Cleared on round complete or cancel.

5. API service layer

Each service is a thin wrapper around the axios client. hospitalId is never sent — it's read from the JWT server-side.

authService

  • POST login({email,password}) → /auth/login
  • POST refresh() → /auth/refresh
  • POST logout() → /auth/logout
  • POST changePassword({current,new}) → /auth/change-password

patientService

  • POST create(payload)
  • GET get(id)
  • PUT update(id, patch)
  • PUT discharge(id, body)
  • GET timeline(id)
  • Plus NoK CRUD scoped to /patients/:id/next-of-kin

vitalsService

  • POST record(patientId, vitals) — response may include escalation object
  • GET list(patientId)
  • GET latest(patientId)

roundService

  • POST create(payload) → /rounds (returns auto-ordered queue)
  • PUT start(id), complete(id), cancel(id)
  • POST addReview(roundId, body)
  • PUT updateReview(roundId, reviewId, body)

Other services follow the same shape — see project structure list for full enumeration. All non-auth services use the shared axios instance and inherit the 401-refresh behaviour.

8. Error handling

ScenarioUX
Network offlinePersistent yellow banner: "You're offline. Changes will not be saved." Retry button. Detected via navigator.onLine + online/offline events.
400 validationMap response fieldErrors to RHF setError. Top-level message in toast.
401 after refresh failauthStore.clearAuth() + redirect to /login?expired=1. Show toast "Session expired".
403Render <Forbidden/> in place. Do not redirect.
404Inline EmptyState ("Patient not found"). No redirect.
409 conflictSpecific message per code: e.g. "A MORNING round is already in progress for this team."
500Generic toast "Something went wrong on our side." + console.error full payload. Sentry hook (if wired).
Kafka delayOptimistic update on the client (push to local list); reconcile on next poll. Mark as "Pending sync" until visible in server response.

9. Polling strategy

  • Custom usePolling(fn, ms): starts on mount, clears on unmount, pauses when tab hidden (document.visibilityState).
  • Ward dashboard: GET /wards/:id/dashboard every 30s
  • Open escalations list: every 20s
  • Current shift status: every 60s
  • All other data: manual refetch (refresh button)
  • Each polled view shows "Last updated: Xs ago" indicator (recomputed every second from a stored timestamp).

10. Responsive design

  • Targets: md (≥768px) tablet workstations and lg (≥1024px) desktop. Mobile (<768px) is explicitly unsupported.
  • Tablet-first design for: Vitals form, Round wizard, Handover entry, Patient detail.
  • Desktop-first design for: Admin pages, Supervisor dashboard, Reports.
  • Sidebar collapses to icon-only at <md (not used in production but graceful).

11. Accessibility

  • Every <input/>, <select/>, <textarea/> wrapped by FormField with associated <label/>.
  • Status colour always paired with a text label or icon (e.g. RED escalation has both pulsing red border and "RED" pill).
  • Focus rings (focus:outline focus:ring-2 ring-blue-500) on every interactive element.
  • Modal: role="dialog" aria-modal="true", focus trapped, ESC closes.
  • Round wizard: ↑/↓ navigate patient queue; Enter opens current patient.
  • RED escalation and NEWS≥7 emit aria-live="assertive" announcement.
  • Tab order follows visual reading order on every page.

12. Design decisions (judgment calls)

  1. Role switcher in sidebar is a prototype-only convenience. In production, role is fixed per session from the JWT.
  2. Vitals form preselects ALERT for LOC because it's the most common state and reduces taps; nurse must explicitly change it for non-alert patients (a deliberate friction at the safe end).
  3. Discharge CONFIRMED gating: option is rendered but disabled for non-consultants (with a tooltip), rather than hidden. This keeps the option discoverable so juniors know it exists and who can set it.
  4. Round wizard remembers position via sessionStorage (per spec) — but on a different patient, the in-progress draft is preserved separately so partial reviews aren't lost.
  5. Optimistic task-state advance: the UI flips status immediately and rolls back on error toast. Clinical staff cannot tolerate a 400ms wait before a button registers.
  6. Patient names shown as LASTNAME, Firstname per UK clinical convention, even though Nigerian convention often differs — consistent format aids quick scanning. (Re-evaluate with end users.)
  7. RED escalation modal blocks the page on submit so the nurse cannot accidentally navigate away before acknowledging the consequence.
  8. Handover sign-off requires all patients to have a note; the supervisor cannot sign off on an incomplete handover. This is stricter than the PRD demands but is the clinically safer default.
